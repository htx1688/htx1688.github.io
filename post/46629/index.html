之前我们讨论了早期创业公司如何建设开发者社区并追踪产品与市场的契合度。不过，如果没有一个强大的开发者关系团队，这些举措很难取得效果。 

开发者关系专员往往会担任信息枢纽的职务，经常要与产品、销售和营销等其他运营团队合作，并与所有人同步信息。许多Web3创业公司都非常重视开发者，因此我们有必要更深入地了解这个岗位。尤其是研究早期创业公司有效管理开发者关系将带来哪些附加值，以及开发者关系为何能决定Web3生态的健康度。

本文重点：

  • 什么是开发者关系？其发展历程是什么？

  • 开发者关系如何在融资轮中推动创业公司向前发展？

  • 开发者关系会在哪些方面为早期创业公司创造价值？

  • 招聘开发者关系岗位时有哪些注意事项？

  • 开发者关系如何推动 Web3 行业向前发展？

什么是开发者关系？

开发者关系（下文简称DevRel）岗位的终极目标是鼓励第三方开发者为某一技术生态开发软件和应用，并为了这一共同目标而展开一系列举措。 

开发者关系岗位的发展简史

要理解DevRel的作用，就必须进一步了解其出现的背景。随着开源文化的兴起，许多人最初在开源社区都做着开发者关系相关的工作，比如推广技术并撰写科普文章。一直到公司开始更积极地面向开发者做销售，DevRel才成为了一个正式的职能岗位。

软件布道者（ software evangelist ）的诞生

据称，苹果公司在上世纪80年代开创了“软件布道者”这一岗位，软件布道者的工作是鼓励开发者为macOS和之后的iOS开发应用。 

苹果意识到，平台的价值必须得由平台上运行的软件支撑。比如，iPhone之所以如此成功，一部分是因为它为消费者提供了一系列丰富又吸引眼球的应用。虽然很多应用最初都是苹果自己开发的，但之后随着苹果生态不断发展，第三方应用占到App Store 的99.99%。

“ 产品驱动增长 ” 以及 “ 开发者优先策略 ” 的兴起

在2016年，OpenView Venture Partners的Blake Bartlett在投资了Optimizely和DataDog等公司后，首次提出了“产品驱动增长”（product-led growth，缩写为PLG）这一概念。“产品驱动增长”背后的假设是，产品的应用量可以实现有机增长，无需从上至下大幅投资于销售或营销。这一模式通常适用于SaaS公司，这类公司的用户可以自己注册并免费试用产品，以此来实现增长。这样做的目的是让用户能够在自己的圈子内推广产品，为产品建立口碑。 

“产品驱动增长”的策略对于那些直接面向开发者的公司来说特别有效，比如Stripe、Twilio和MongoDB。这些公司采用DevRel策略来加速增长。在这种情况下，DevRel就显得十分重要，因为产品的用户界面设计问题导致开发者无法获得最佳的使用体验。由于缺乏流畅的前端用户体验，开发者无法轻松使用新的API或SDK，因此就需要更多指导和帮助。  

所以DevRel在这里就发挥了重要作用，他们让开发者对软件产生一个好的第一印象，并在每个环节为开发者提供良好的开发体验（注：开发者体验也称DX）。 

DevRel 的几大职能

像Twilio或Atlassian这类开发者优先的公司会建立完备的DevRel部门，并将岗位划分成各个DevRel职能，比如开发者营销或社区管理。而对于规模相对较小的创业公司来说，这些职能通常由一个人负责，这个人的头衔中也许甚至没有“DevRel”这个词。

Web3 的开发者关系

在Web3行业，DevRel的职责和目标与Web2非常相似。无论是Web2还是Web3，DevRel专员都要对接不同的开源社区，而每个社区都有自己独特的文化。Web3 DevRel专员所面临的挑战是没有合适的工具来有效追踪各项衡量指标并简化低级别的开发任务。比如，Web2前端开发者可以使用React或Vue.js等专门的成熟工具，因此可以大幅缩减开发web应用的时间。而在Web3行业，此类框架并不成熟，因此Web3教育者需要投入更多精力来支持这些新进开发人员。

Web3 DevRel 仍处于发展初期

虽然一些Web3企业建立了完备的DevRel部门，比如Consensys、Alchemy和Chainlink Labs，但这类企业目前仍是凤毛麟角。大多数Web3创业公司只有一名DevRel代表，或甚至没有设立这个岗位，而是将DevRel相关职能临时分配到了不同团队成员头上。

即使Web3团队设立了多个DevRel岗位，这些员工通常都没有太多Web3的工作经验。许多人都是在过去12个月内从谷歌、Facebook和亚马逊这些Web2大企业转行过来的。而Web2 DevRel社区也开始注意到了这个趋势：2022年1月，DevRelCon创始人Matthew Revell预测未来一年中仍会有大量优秀的开发者关系从业者跳槽到区块链公司。

因此，大多数DevRel专员仍在摸索阶段，并且在努力建立一套最佳实践，可以为具体的项目目标提供支持，而不是单纯地整合专业知识。

DevRel 对各类 Web3 企业来说意义重大

在讨论Web3开发者社区时，我们简要谈到了需要建立开发者社区的几种项目类型。由于开发者社区建设属于DevRel的职责范畴，因此公司如果想要建设开发者社区，就必须了解DevRel的工作机制。

需要建设开发者社区的公司通常分为三大类，并且这三类公司的业务目标都十分相似。

Web3 基础设施

这类公司往往不会绑定某一个协议，而是会开发各种工具和技术，提升Web3生态的互操作性。

  • 数据预言机

  • L2扩容方案

  • 与区块链交互的API接口

  • KYC管理平台

  • 分布式文件系统

  •  区块链应用SDK和中间件

DevRel 的业务目标： 提高公司技术的集成数量，通过对API调用和智能合约交互收费来获得收入。

L1 区块链和 DeFi 协议

通常由去中心化团队来管理并维护区块链技术，而开发者教育则是由开源基金会负责，比如以太坊基金会或Cardano基金会。

DevRel 的业务目标： 提高接入协议的项目数量，通过交易费来增长收入，并将收入用于为矿工和节点运营商提供经济激励，以保障网络安全。

交易平台和市场

这类通常是中心化的数字资产交易平台。虽然这类平台主要针对终端用户，但也会为开发者提供API接口和SDK，以自动执行一部分交易操作。  

DevRel 的业务目标： 提高集成数量，以此提升平台的交易量，通过收取交易费获得收入。许多协议也提供免费和付费的聚合交易数据，开发者可以在自己的dApp中集成这些数据。

如果Web3创业公司希望在以上任何一个类别建立市场份额，都需要尽早投入DevRel。通常来说，创业公司在pre-seed轮都会聚焦产品开发。而一旦获得融资后，DevRel就显得至关重要，而且会为投资者建立信心。 

DevRel 如何帮助创业公司进行融资

创业公司如果能尽早掌握DevRel的要领，将更有可能更快地完成各轮融资。Heavybit（一家VC，专门投资开发者优先的创业公司）的普通合伙人Dana Oshiro在一篇名为“开发者社区的防御能力和价值”的演讲中提到，DevRel专员可以激励投资者继续为创业公司提供融资。因为DevRel可以为投资者提供他们感兴趣的各种数据和指标。 

Oshiro将融资分成以下几个阶段，并阐述了各个阶段DevRel的职责：

种子轮

在pre-seed或冷启动阶段，DevRel的职能通常会临时或随机地分配给创始团队的各个成员。团队里最擅长演讲的开发者会负责演示产品；一名联合创始人会负责编写文档；另一名联合创始人会负责开发者营销。 

  • 目标： 在这个阶段，潜在投资者重点关注的是创始人能否验证产品与市场的契合度，以及是否由足够多的早期用户来维持产品发展。

  • 目标和衡量指标： 投资者主要感兴趣的指标是早期参与度和关注度，比如网站流量、社交媒体活跃度、社区注册人数、demo演示次数以及开发者反馈数量等。

  • 重点总结 ：如果DevRel职能被分配给多个团队成员，那么创业公司就必须集中地协调、追踪并量化关键领域的DevRel工作。这样，在拉投资的时候就可以提高可信度，并在之后几轮的融资中也可以起到推动作用。

A 轮融资

在这个阶段，必须要设立专门的DevRel岗位。创业公司在这个阶段会面临更严格的审核标准，因为投资者需要确保企业能够在钱烧光之前达成下一个里程碑。因此就需要一名全职员工来推出DevRel计划并追踪效果，无法再让创始人或开发负责人来兼职。

  • 目标： 投资者关注的是创业公司是否能招募并留住开发者，并且开始对标成功的项目。

  • 目标和衡量指标： 投资者希望公司能够达成以下目标：（注：这些目标都有具体的衡量标准，比如开发者交互次数、推荐顾客数量以及留存的社区成员数量）

  • 建立产品和社区认知。

  • 招募开发者，激励开发者首次调用API接口或智能合约。 

  • 分流和转化，获得产品反馈。

  • 重点总结 ：DevRel专员的作用至关重要，可以负责建立标准化流程，搜集投资者感兴趣的指标，并帮助开发者顺利进入生态。

A 轮融资后  

在这个阶段，只靠一个人担任DevRel岗位可能就不够了，这个人可能会忙得焦头烂额。公司要着手将DevRel岗位分解成多个职能，比如开发者社区经理、开发者营销负责人或者按地区来划分，比如针对亚洲地区或英语地区的开发者倡导者。

  • 目标： 建立防御能力。在这个阶段，投资者关注的是创业公司是否能招募并留住开发者，而且有足够多成功的开发者可以作为参考和案例。

  • 目标和衡量指标： 在千万美元级别的融资轮中，投资者希望看到更细致的数据，能够让他们深入了解创业公司的发展势头，确保公司可以：

  • 继续维持知名度 和获客 方面的势头，并基于产品反馈 进行迭代优化。

  • 留住 真正具有用户粘性的忠诚开发者。

  • 鼓励开发者将技术推荐 给别人，并自发进行宣传推广。

  • Github fork、集成、集成数量、生态中的第三方应用以及合作伙伴数量等都是重要的衡量指标。

  • 重点总结 ：创业公司需要建立DevRel团队，每个成员可以负责不同的职能并达成每一个关键领域的衡量指标，比如生态增速以及产品参与度等。

当然，无论在什么阶段，收入都是一个关键的衡量指标，但收入并非DevRel的主要职责。不过DevRel人员必须理解他们开展的计划对创业公司的财务健康度会产生哪些直接影响。有时这点很难做到，因为一些计划很难与收入联系起来，并且需要花费很长时间才能看到结果，特别对于早期阶段的创业公司来说更是如此。

早期创业公司应该关注哪些 DevRel 职能？

许多开发者优先的创业公司都会将开发者赋能和社区发展放到第一位。这并不意味着公司会忽视营销或倡导职能，只是仍然会将这些职能分散在团队中。而开发者赋能和社区职能会成为DevRel专员的全职工作。

为什么会更重视开发者激活而非开发者认知度？

首先要记住，这只是某个阶段的一个片段，而非永恒的状态。随着创业团队不断扩张，重心将重新偏移到开发者认知度上。但在早期冷启动的阶段，创业公司往往更擅长建立认知度，而不是留住用户。

特别是对Web3项目来说尤其如此，因为创始人往往都面临巨大压力，需要建立网络效应以吸引投资。重点是要让大家对于其总体的产品愿景和路线图产生兴趣。

但是有无数个项目都在竞争开发者的时间和注意力。因此光有一个吸引人的愿景是不够的。如果开发者体验很差或社区反馈不够及时，这些都会快速消磨开发者的兴趣，最终他们会迁移到其他体验更好的项目上。如果只能建立开发者认知度而无法留住开发者，那一切都是徒劳。

我们在之前的文章中已经提到如何建设开发者社区了，因此本文的重点是开发者赋能。

提升开发者赋能的策略

大多数开发者优先的公司都会采取许多类似的开发者赋能策略。然而很多创业公司往往会无视这些策略，或者不会认真执行。开发者赋能策略包括以下几种：

  • 开发优质的技术文档、教程和入门指南。

  • 举办面向产品的网络研讨会、编程直播课程以及教学视频。

  • 制定清晰的错误提示，包含产品架构中每一层的问题。

创业公司往往无法有效执行这些赋能策略，因为他们并没有想到要在端到端构建全面的开发者体验。这些赋能策略有时也被称为“开发者体验设计”。

开发者体验设计的重要意义

下图展示了如何进行开发者体验设计。这是SenGrid的开发者体验设计，SenGrid是交易和营销电子邮件自动化平台，目的是让开发者在旅程中一步步成为活跃的忠诚用户。

Todd Moy的流程图——资料来源。

这个例子来自名为《开发者路径解析》的案例研究，该案例由产品设计师Tod Moy发布。这是一个非常具有说服力的案例，因为它覆盖了整个开发者旅程，涵盖了内容和产品触点。Moy研究了一个名为“Dewey”的开发者的旅程，并试图记录了他的动机、痛点和亮点。 

这类旅程通常是由产品团队制定的，但如果产品的核心不是用户接口，难度就比较大了。开发者优先的公司往往会非常重视DevRel，这类公司会推动制定开发者旅程，并精准归纳出开发者“人格”。由于在早期Web3创业公司中几乎没有产品经理岗位，因此Web3 DevRel专员就显得格外重要。他们会负责用户调研和产品优化工作。

Web3 创业公司如何招聘适合的 DevRel 人才？

在Web3行业，各公司对DevRel人才的竞争十分激烈。2022年的《TrueUp Crypto岗位调研报告》发现，Web3对于社区和开发者关系岗位的需求是其他技术行业的四倍。这意味着创业公司需要灵活设置他们的招聘需求。

在非核心领域适当放宽要求

由于DevRel人才非常抢手，因此创业公司要灵活设置岗位需求，特别是在Web3经验和工作地点上适当放宽要求。

  • 工作地点：由于大多数Web3组织都高度地去中心化，因此Web3创业公司可以在工作地点上轻松做出妥协。唯一需要注意的是，在一些司法管辖区招聘员工可能会面临一些政策阻碍；另外，必须要保障各地员工所在的时区重合度足够高，以便高效协调。

  • Web3的工作经验：许多Web3岗位描述都要求求职者要有一定Web3工作经验。其实，求职者只需要有Web2行业的技术经验就够了，因为Web3的技术经验可以在工作中慢慢累积。另外，如果求职者会对Web3岗位感兴趣，他们很可能已经掌握了智能合约、NFT和去中心化等Web3的基础概念。

然而，许多创业公司即使在这两点上放宽了条件，仍然很难招到合适的DevRel人才。如果是这样的情况，创业公司可以进一步放宽条件，考虑从其他相关学科跨界过来的人才。

寻找拥有相关技能的求职者

不一定非得要招聘有过DevRel从业经验的人才。因为这是一项新兴职业，所以有很多人都是从其他相关领域跨界过来的。比如，创业公司可以考虑招聘一名技术写作人员，在Discord上回答技术问题；招聘一名社区经理，拥有技术知识并能够写文档；或者招聘一名开发者，对DevRel岗位感兴趣，并且擅长技术沟通。 

对于多数创业公司来说，早期招聘的DevRel人员都是什么都可以做一点的全才。但要记住一点，几乎没有人能在营销、赋能、倡导和社区方面样样都行。因此，你要找的人才必须要能够满足你最核心的需求。

重视共情能力

无论求职者的专业背景如何，DevRel岗位一个不可或缺的能力就是共情能力。这需要一个人能够克服自己内心的偏见和既有知识。他们需要去了解目标用户的动机和知识缺口。创业公司如果可以理解不同类型开发者的诉求，就可以更好地定制化沟通和产品策略，以满足目标开发者的需求。

如何评估共情能力

评估共情能力比评估技术能力更难，因此在面试的时候共情能力往往会被忽视。公司会问一些常见的问题，以评估求职者的社交能力，但DevRel的任务是真正站在开发者的角度看问题。

有个办法是让求职者从“五个维度”向不同类型的群体解释某个Web3技术概念（注：“五个维度”源自Wired上非常受欢迎的系列视频）。比如，Chainlink开发者倡导者Patrick Collin在一篇名为《为什么要Web3》的文章里解释了“信任最小化协议”的概念，并将其比喻成两人拉钩后做出的“坚不可摧的承诺”。他认识到普通人仍不熟悉信任最小化的概念，因此用简单通俗的语言解释了这个概念，让所有人都能看懂。深入浅出地解释某个生僻的概念，这是所有DevRel人员都必须具备的技能。

另一个办法就是给求职者规定一个情境，比如一名开发者遇到了技术问题，并在社区中寻求帮助。求职者需要在这个情境中扮演DevRel人员来帮助他。这个任务的目的是考察求职者多快可以明确开发者的根本诉求。这里要强调的是，这个任务的关键不是钻研具体的技术细节，而是要了解问题发生的背景情况。 

Web3 为什么需要更多 DevRel 人员？

能够与开发者共情的DevRel团队可以很好地为新进开发者提供指导，帮助他们很快上手复杂的技术生态。对于许多刚进入Web3的开发者来说，Web3更像是一个个城邦而非统一协同的行业，每个城邦的工会都在互相竞争。随着互操作性逐渐成为主流，这种情况也有所改变。然而很多时候仍然很难统一语言和术语。这种分裂不仅体现在语言层面，比如现在有各种不同的智能合约语言；还体现在概念层面，比如现在有各种互相竞争的共识机制、协议、跨链桥、L1和L2等。因此，DevRel专员要在其中扮演翻译的角色，将生僻的技术概念翻译成大家更熟悉的语言。由于Web2的开发工作与Web3差异非常大，因此必须简化复杂的技术概念才能吸引更多Web2开发者跨界到Web3。

DevRel岗位的独特之处在于他们需要在两方面保持忠诚度。一方面，DevRel人员的首要职责就是确保项目取得成功。而另一方面，他们也需要对行业中的开发者社区保持忠诚。无论这些开发者是不是他们的直接客户，他们都希望看到开发者获得成功。这种愿景激励着他们创作更多科普文章，并在Stack Overflow等开发者论坛上回答问题。这样做可以有效提升公司的品牌声誉，但这并不是驱动他们的根本目标。

更重要的是，Web3基础设施创业公司的成功将越来越离不开DevRel人员。Web3基础设施领域正在不断发展，创业者也意识到市场对于去中心化基础设施解决方案的需求正在不断上升。而这意味着竞争正在白热化。在同一垂直领域的创业公司将越来越需要DevRel专员，以在竞争中脱颖而出，并提高在开发者中的市场占有率。无论谁最后胜出，如果Web3创业公司能够正视DevRel的价值，这都将使整个Web3行业受益。
